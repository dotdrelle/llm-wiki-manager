# Repository Guide

## Goal

`llm-wiki-manager` orchestrates multiple `llm-wiki` workspaces from one local
control point. It wires Docker Compose services, workspace-specific `.env`
files, MCP endpoints, the Confluence exporter, the production job agent, and an
external mailer MCP endpoint.

## Architecture

- `wiki-workspace`: main operator CLI for workspace creation, service startup,
  Docker Compose commands, port allocation, and local MCP auth token generation.
- `docker-compose.yml`: shared compose stack for wiki UI, wiki MCP,
  workspace-scoped `agent-cme`, and `agent-wiki-production`.
- `.env.example`: manager-level settings, including external transverse MCP
  endpoint URLs and local bearer tokens.
- `workspaces/.env.example`: per-workspace settings generated by
  `./wiki-workspace config`.
- `SKILL.md`: agent workflow for the Confluence to wiki to deliverable pipeline.
- `workspaces/`: local workspace config and symlinks. This directory is
  gitignored and must not be treated as source.

## Constraints

- The manager does not implement `llm-wiki`, `agent-cme`,
  `agent-wiki-production`, or `agent-mailer-api`; it configures and starts their
  published images or local services.
- Keep the normal pipeline as `ingest`, `build`, `export`, then `polish`.
  The legacy `copy` step is only for deployments that explicitly configure
  import mappings.
- Do not commit generated workspace `.env` files, `.cme` state, exported raw
  documents, wiki output, or local symlink targets under `workspaces/`.
- MCP auth tokens are local coordination secrets for internal local calls. They
  are not MailerSend, Atlassian, OpenAI, Anthropic, or model-provider API keys.
- Keep provider credentials inside each workspace `.wikirc.yaml` or the owning
  service environment. Do not move third-party secrets into the manager `.env`.
- Avoid hard-coded absolute paths in docs and examples. Use placeholders such as
  `<absolute-path-to-workspace>` and `<generated-local-token>`.

## Common Commands

```bash
./wiki-workspace config <workspace> [path]
./wiki-workspace up <workspace>
./wiki-workspace list
./wiki-workspace wiki <workspace> doctor
./wiki-workspace wiki <workspace> ingest
./wiki-workspace wiki <workspace> build
./wiki-workspace wiki <workspace> export
./wiki-workspace cme <workspace> up
./wiki-workspace mailer status
```

Docker Compose commands should normally go through `wiki-workspace` so the
correct workspace `.env`, project name, ports, and volume mounts are used.
